# Convert Agent Guide

## Project Summary
- `Convert` is a static React + Vite unit converter app for browser-only usage.
- The codebase is registry-driven: routes, metadata, search behavior, and most page content fan out from shared data files instead of one-off page definitions.
- Current code-derived scope on April 4, 2026: `132` converters across `28` categories. Re-check from code before repeating counts in docs or release notes.

## Source Of Truth
- Treat [`src/data/converterRegistry.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/converterRegistry.js) as the canonical converter catalog.
- Treat [`src/data/converterRoutes.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/converterRoutes.js) and [`src/data/categoryGroups.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/categoryGroups.js) as the routing and navigation layer.

- Treat generated files in [`public/`](/Users/pruthvikauticwar/Vibe_Code/Convert/public) as outputs. `public/sitemap.xml` is regenerated by the build.

## Working Style
- Read the registry and handler flow before making feature edits. Most changes should be data-driven instead of page-by-page patches.
- Prefer extending shared patterns in [`src/pages/ConverterPage.jsx`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/pages/ConverterPage.jsx) and shared components before introducing bespoke UI.
- Keep the app static-first. Avoid adding server dependencies unless there is an explicit product decision to do so.
- Verify claims in docs against code. `README.md`, `PROJECT_PLAN.md`, and `TESTING_REPORT.md` have drifted before, especially around total route counts and the mention of currency/live-data support.

## When Adding Or Changing A Converter
1. Update registry metadata, aliases, examples, and validation copy in [`src/data/converterRegistry.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/converterRegistry.js).
2. Update any routing or category metadata in [`src/data/converterRoutes.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/converterRoutes.js) or [`src/data/categoryGroups.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/data/categoryGroups.js).
3. Add or adjust handler behavior in [`src/lib/converterHandlers.js`](/Users/pruthvikauticwar/Vibe_Code/Convert/src/lib/converterHandlers.js) and supporting math/parser utilities.
4. Update tests in [`tests/`](/Users/pruthvikauticwar/Vibe_Code/Convert/tests) and add or adjust Playwright coverage in [`e2e/`](/Users/pruthvikauticwar/Vibe_Code/Convert/e2e) when user-visible flows change.
5. Run the verification commands below.
6. Sync docs if counts, categories, capabilities, or launch claims changed.

## Verification
- Quick verification: `npm run verify`
- Full verification: `npm run verify:full`
- Use full verification when changing search behavior, routing, shared converter UX, or any flow that affects multiple pages.

## Product Prioritization
- Favor converters or calculators with clear search intent, repeat usage, or adjacency to existing categories.
- Avoid adding niche tools just to increase counts. A smaller set of well-polished, searchable tools is better than bloated surface area.
- For new categories, check whether the category earns its keep in navigation, search, and SEO before expanding it.
- Use [`MARKET_RESEARCH.md`](/Users/pruthvikauticwar/Vibe_Code/Convert/MARKET_RESEARCH.md) before committing to new categories or large expansion batches.

## Review Priorities
- Broken registry invariants
- Handler mismatches between metadata and runtime behavior
- Search regressions
- Invalid examples or validation copy
- Docs that make claims the code does not support
